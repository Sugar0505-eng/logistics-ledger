## Context

全新绿地项目，目标是一个**纯本地、无后端、无账号**的移动端记账 App，供物流平台内部人员使用。首期仅交付 Android（开发机本身即可编译、调试并直接分发 apk），但选型上保证 iOS 后续可零业务代码改动地扩展。核心约束：

- 离线可用（仓库/港口网络不稳定），OCR 必须端上运行。
- 数据只在本机，不上云，不做账号体系。
- 涉及金额计算，需避免浮点精度问题。
- 集装箱柜号遵循 ISO 6346，可用规则校验提升识别可靠性。

## Goals / Non-Goals

**Goals:**
- 单一 Flutter 代码库，跨平台可移植，首期产出可用的 Android apk。
- 三层数据模型（账目记录 ▷ 账单 ▷ 额外费用）+ 独立的车牌库、费用预设。
- 端上 OCR 识别柜号，含 ISO 6346 提取/校验与用户确认。
- 按单个账目记录导出 CSV，额外费用动态成列。

**Non-Goals:**
- 不做 iOS 打包/分发（仅保持代码可移植）。
- 不做云同步、多端共享、用户账号、权限体系。
- 不做票据/发票图像的全字段识别（OCR 仅服务于柜号）。
- 不做财务报表分析、统计图表（首期仅录入与导出）。

## Decisions

### 决策 1：框架选 Flutter（而非 React Native）
- **理由**：端上 OCR 的官方方案 Google ML Kit 在 Flutter 生态（`google_mlkit_text_recognition`）成熟、离线、免费、无次数限制；自绘 UI 两端一致；单库出双端。
- **备选**：React Native（ML Kit 需社区桥接，集成略繁）；原生双端（开发量翻倍，违背跨平台诉求）。

### 决策 2：本地存储用 SQLite，经由 `sqflite`（实现期修订）
- **原定**：使用 `drift`（类型安全、编译期校验）。
- **修订为 `sqflite`**：实现阶段本机未安装 Flutter，采用"先写源码后构建"策略。`drift` 依赖 `build_runner` 代码生成，而代码生成只能在装好 Flutter 后运行、当下无法验证；`sqflite` 为纯手写 SQL，所写即最终代码、可直接人工核对，无代码生成步骤，更契合当前约束。
- **理由**：三层关系结构（外键、级联删除、聚合求和）用关系型数据库最自然；`sqflite` 轻量、稳定、无构建期依赖。
- **代价**：手写 SQL 与 Map 映射，失去编译期 schema 校验（以单元测试与人工核对弥补）。
- **备选**：`drift`（更安全但需 codegen，当前不可验证）；`Hive`/NoSQL（关系建模与聚合查询不便）。

### 决策 3：金额用整数分存储
- **理由**：避免二进制浮点精度误差。数据库与计算层一律以"分"（整数）存储与运算，仅在 UI 显示与 CSV 输出时格式化为小数。
- **备选**：`double`（有精度风险，财务场景不可取）；`Decimal` 库（可行但存储仍建议整数分）。

### 决策 4：柜号识别为"OCR + 规则"两段式
- **流程**：ML Kit 全图文字识别 → 正则提取 `[A-Z]{4}\d{7}` 候选 → ISO 6346 校验码验算 → 弹确认框（展示是否通过校验）→ 用户确认/修改后写入。
- **理由**：OCR 不可全信，规则提取+校验+人工确认三道关把准确率与可控性拉满。
- **备选**：直接信任 OCR 文本（易错）；云端 OCR（违背离线约束）。

### 决策 5：CSV 动态列由"全账目费用名并集"驱动
- **流程**：导出前扫描该账目记录下所有账单的额外费用名称，去重排序得到列集合；逐行填值，缺失留空；每行追加小计列。
- **理由**：满足"每种费用单独成列"且列数随数据自适应，Excel 打开即规整报表。

### 决策 6：分层架构
- UI 层（页面/组件）→ 业务/状态层（如 Riverpod/Provider）→ 数据层（drift DAO）→ 能力封装（OCR service、CSV exporter、文件分享）。OCR、CSV、文件分享均封装为独立 service，便于替换与测试。

## Risks / Trade-offs

- **OCR 识别率受照片质量影响** → 规则提取 + 校验码标识 + 强制用户确认框兜底；未识别时无缝转手动输入。
- **首期不出 iOS，未来出包仍需 Mac/苹果账号** → 通过坚持 Flutter + 不使用 Android 专有 API 保留可移植性；iOS 出包问题留待后续（可用 Codemagic 云构建）。
- **删除费用预设可能与历史账单同名费用混淆** → 额外费用名称在账单中以快照值独立存储，不外键引用预设；预设仅为录入便利。
- **金额整数分方案要求全链路一致** → 在数据层与 service 层统一封装"分↔元"转换，UI 与 CSV 仅在边界格式化。
- **大账目记录导出性能** → 首期数据量小（内部使用），暂不优化；如需可后续流式写文件。

## Resolved Decisions

- **日期粒度**：仅到天，统一格式 `yyyy-MM-dd`，不含时间/时区。
- **CSV 编码**：输出 UTF-8 **含 BOM**，确保中文在 Excel 中正确显示。
- **录入时新建车牌**：账单录入页的车牌选择器支持"新建车牌并入库"，新建车牌即写入车牌库（受唯一性校验约束）后选中。
